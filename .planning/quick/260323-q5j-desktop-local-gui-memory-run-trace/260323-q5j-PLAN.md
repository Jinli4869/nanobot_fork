# Quick Task 260323-q5j Plan

**Description:** 执行 desktop/local 真实 GUI memory 命中测试并分析 run trace。
**Date:** 2026-03-23
**Mode:** quick

## Objective

基于 `260323-q1s` 已确定的任务与判定标准，实际执行一次宿主机 `--backend local` GUI run，记录真实产生的 run artifact，并输出 memory 是否命中、命中到了哪些条目、以及后续 GUI step 是否体现了这些 memory。

## Constraints

- 这次 quick task 必须执行真实 desktop/local GUI 测试，不重新设计测试方案。
- 只能产出文档 / state artifact，不改业务代码，不覆盖任何现有未提交修改。
- 执行结果必须明确记录：
  - 实际执行命令
  - 实际观察到的 run 根目录
  - 实际观察到的 trajectory trace / attempt trace 路径
  - memory-hit outcome（pass / partial / fail）

## Context

- 参考上一条 quick 设计结论：`260323-q1s-PLAN.md` 与 `260323-q1s-SUMMARY.md` 已把主任务固定为 `切换回浏览器，并重新打开刚刚关闭的标签页。`
- `opengui/cli.py` 的 local CLI 默认从 `~/.opengui/memory` 加载 memory，并把每次 run 写到仓库相对目录 `opengui_runs/`。
- `opengui/agent.py` 在 run 开始时调用 `_retrieve_memory(task)`，命中证据写到 run 级 trajectory trace；attempt 级别动作和结果写到各 attempt 目录下的 `trace.jsonl`。

## Tasks

1. **冻结执行前基线并确认 memory 前置条件**
   - 记录当前工作树状态，证明本次 quick task 不触碰未提交代码；只允许新增 `.planning/quick/260323-q5j-desktop-local-gui-memory-run-trace/` 下文档和必要 state 更新。
   - 执行前确认本机具备可用 `~/.opengui/config.yaml` embedding 配置，以及 `~/.opengui/memory/` 下已有 OS / APP memory 文件；若缺失，停止并把阻塞原因写入结果文档，不做替代实现。
   - 把本次唯一测试命令固定为：
     - `python -m opengui.cli --backend local --task "切换回浏览器，并重新打开刚刚关闭的标签页。"`
   - 同时把人工前置状态写入结果文档：
     - 浏览器不在前台
     - 浏览器中存在“刚刚关闭的标签页”可恢复

2. **执行真实 desktop/local run 并捕获实际 artifact 路径**
   - 在宿主机桌面按前置状态准备后，运行固定命令，保留 stdout/stderr 关键结果。
   - 执行结束后定位最新 run 根目录，并在结果文档中写入观察到的精确路径：
     - run 根目录：`opengui_runs/<observed_run_stamp>/`
     - trajectory trace：`opengui_runs/<observed_run_stamp>/trace_<observed_timestamp>.jsonl`
     - attempt trace：记录所有实际出现的 `<task_slug>_<epoch_ms>_<attempt>/trace.jsonl`，至少写出 `_0`，若出现 `_1`、`_2` 也一并列出
   - 如果命令报错、人工干预、或 run 目录未生成，仍要记录失败现场，不得用推测路径代替实际观察结果。

3. **分析 trace 并输出 memory-hit 结论**
   - 读取 trajectory trace，查找 `memory_retrieval` 事件，逐项记录：
     - `hit_count`
     - 命中的 `entry_id`
     - 命中的 `memory_type` / `platform` / `app`
     - `context` 是否非空
   - 读取 attempt trace，按顺序总结：
     - 是否出现 `attempt_start`
     - 每个 `step` 里是否体现应用切换热键、浏览器恢复标签页热键，或与之明显等价的操作
     - 最终 `attempt_result.success` 与错误摘要
   - 产出单一结果文档，明确给出以下之一：
     - `memory-hit outcome: pass`：同一次 `memory_retrieval` 同时命中 OS + browser memory，且 attempt trace 显示相应热键思路
     - `memory-hit outcome: partial`：memory 命中了，但 GUI 执行未完成或仅部分体现 memory
     - `memory-hit outcome: fail`：未命中预期 memory，或 trace 无法证明命中链路成立
   - 在结果文档末尾补充一句可复核结论：下一位执行者无需重新跑测试，仅根据文档列出的命令、run 目录和 trace 路径即可复查。
