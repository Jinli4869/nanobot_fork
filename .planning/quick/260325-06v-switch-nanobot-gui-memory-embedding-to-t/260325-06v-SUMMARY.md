# Quick Task 260325-06v: Align Nanobot GUI Memory Embedding With OpenGUI CLI

**One-liner:** Nanobot GUI memory retrieval now uses the same direct OpenAI-compatible embedding path as OpenGUI CLI for custom providers, fixing DashScope-compatible memory indexing in Telegram/Nanobot runs.

## What Changed

- `nanobot/agent/tools/gui.py`
  `GuiSubagentTool._build_embedding_adapter()` now prefers the provider's direct OpenAI-compatible client when one exists, instead of routing embeddings through LiteLLM.

- `nanobot/agent/tools/gui.py`
  Added `_normalize_direct_embedding_model()` so values like `dashscope/text-embedding-v4` are normalized to `text-embedding-v4` before calling the direct embedding endpoint.

- `tests/test_opengui_p6_wiring.py`
  Added regression coverage proving the direct client path is used and that provider-prefixed embedding model names are normalized correctly.

## Verification

- `uv run pytest tests/test_opengui_p6_wiring.py tests/test_opengui_p8_trajectory.py tests/test_opengui_p3_nanobot.py tests/test_opengui_p11_integration.py`
- Result: `54 passed`

## Config Guidance

For `~/.nanobot/config.json`, keep GUI embedding config aligned with the OpenGUI CLI values:

```json
"gui": {
  "backend": "local",
  "background": true,
  "displayWidth": 1440,
  "displayHeight": 900,
  "artifactsDir": "gui_runs",
  "maxSteps": 15,
  "skillThreshold": 0.6,
  "embeddingModel": "text-embedding-v4"
}
```

Your `providers.custom.apiBase` should continue pointing at the DashScope-compatible endpoint, and `providers.custom.apiKey` should remain the key used for both chat and embeddings.
