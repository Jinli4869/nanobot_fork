# Quick Task 260323-q1s Plan

**Description:** 设计 desktop 真实 GUI memory 命中端到端测试并标明应查看的 run trace。
**Date:** 2026-03-23
**Mode:** quick

## Objective

设计一条面向 `--backend local` 的真实桌面 GUI memory 命中验证方案，优先证明同一次 run 同时命中 browser app memory 和 macOS OS memory，并明确执行后应该检查哪些 run artifact / trace 文件、看哪些字段才算命中。

## Context

- `opengui/cli.py` 的 desktop/local CLI 路径默认从 `~/.opengui/memory` 加载 memory，并把每次 CLI 执行写到仓库相对目录 `opengui_runs/<timestamp>/`。
- `opengui/agent.py` 会在 run 开始时先做一次 `_retrieve_memory(task)`，把命中的条目记录到 trajectory trace 的 `memory_retrieval` 事件里；每个 attempt 还会在 `<attempt_dir>/trace.jsonl` 里记录 `attempt_start`、`step`、`attempt_result` 等事件。
- 当前实际 memory 原文里，OS 侧是 `~/.opengui/memory/os_guide.md` 的 macOS 快捷键，browser 侧是 `~/.opengui/memory/app_guide.md` 的 browser hotkeys，因此测试任务必须同时显式触发“切换应用”和“浏览器内部热键”这两类意图。
- 最近 quick fix `260323-p01` 已经改善 GUI action 的轻微格式偏差恢复能力，因此这次设计应优先选一个需要热键而非精细坐标点击的任务，降低噪声。

## Tasks

1. **确定单条主验证任务与人工前置状态**
   - 主任务固定为：
     - `切换回浏览器，并重新打开刚刚关闭的标签页。`
   - 选择这条任务的原因：
     - “切换回浏览器”直接对应 macOS OS memory 中的应用切换知识。
     - “重新打开刚刚关闭的标签页”直接对应 browser app memory 中的 tab restore 热键知识。
     - 两段意图在一次 desktop/local run 内连续发生，最容易在同一次 `memory_retrieval` 里同时召回两类条目。
   - 执行前人工准备状态：
     - 在 macOS 本机前台先切到一个非浏览器应用，确保浏览器不在前台。
     - 浏览器里预先制造“刚刚关闭一个标签页”的状态，避免 agent 即使命中 memory 也无事可做。
     - 不要使用 `adb`、`--dry-run` 或 `--background`；目标就是宿主机真实桌面上的 local backend。

2. **定义执行方式与 run 目录定位规则**
   - 推荐执行命令写进测试说明，保持 desktop/local 路径最短：
     - `python -m opengui.cli --backend local --task "切换回浏览器，并重新打开刚刚关闭的标签页。"`
   - 如果本地配置依赖 embedding provider 才会启用 memory，则明确要求使用已有可工作的 `~/.opengui/config.yaml`，因为 `config.embedding is None` 时不会构建 `MemoryRetriever`，这次测试会直接失去 memory 验证意义。
   - 执行后先在仓库根目录下定位最新 run 根目录：
     - `opengui_runs/<YYYYMMDD_HHMMSS_microseconds>/`
   - 重点查看两个位置：
     - trajectory trace: `opengui_runs/<run_stamp>/trace_<YYYYMMDD_HHMMSS>.jsonl`
     - 首次 attempt trace: `opengui_runs/<run_stamp>/<task_slug>_<epoch_ms>_0/trace.jsonl`
   - 如果发生 agent 外层 retry，再顺序看 `_1`、`_2` attempt 目录，但主验证优先看 `_0`，因为 memory retrieval 发生在 run 级别而不是每个 step 里重新检索。

3. **定义 trace 检查项与通过标准**
   - 在 run 根目录的 trajectory trace 里，先查 `type == "memory_retrieval"` 事件；至少满足：
     - `hit_count >= 2`
     - `hits` 中包含一条 `entry_id == "os-guide-macos-shortcuts"` 或等价的 macOS OS guide 条目
     - `hits` 中包含一条 `entry_id == "app-guide-browser-hotkeys"` 或等价的 browser app guide 条目
     - 两条命中分别带有 `memory_type` 为 OS / APP 类别、`platform == "macos"`，且 browser 条目 `app == "browser"`
     - `context` 非空，说明命中内容已真正注入 prompt，而不是只做了内部 search
   - 在 attempt trace `trace.jsonl` 里，按时间顺序检查：
     - `event == "attempt_start"` 存在，确认该 trace 属于本次 GUI attempt。
     - 至少有一个 `event == "step"` 的 action 显示 agent 在尝试热键型操作。
     - 最理想的模式是先出现应用切换热键，再出现浏览器 tab restore 热键；允许键名大小写或 `ctrl`/`control`、`cmd`/`command` 等轻微表述差异。
     - 若最终 `attempt_result.success == true`，额外确认浏览器确实回到前台且被恢复的标签页重新出现。
   - 结果解释写清楚：
     - `memory_retrieval` 同时命中 OS + browser，且后续 step 使用相应热键思路，即可判定“memory 命中链路成立”。
     - 如果命中了两类 memory，但 GUI 没成功完成，也要单独记录为“retrieval pass / execution fail”，因为这次 quick task 的首要目标是验证 memory 注入和 trace 可观测性。
     - 如果 trajectory trace 根本没有 `memory_retrieval` 事件，优先回查配置是否启用了 embedding，而不是先怀疑 task phrasing。
