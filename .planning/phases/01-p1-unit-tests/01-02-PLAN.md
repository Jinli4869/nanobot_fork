  <name>Task 1: Write SkillLibrary CRUD, search, and deduplication tests</name>
  <files>tests/test_opengui_p1_skills.py</files>
  <behavior>
    - SkillLibrary.add() stores skill, .get() retrieves it, .count returns 1
    - SkillLibrary.remove() deletes skill, .count returns 0
    - SkillLibrary persists to disk: new SkillLibrary(same_path) reloads skills
    - SkillLibrary.list_all() returns all stored skills
    - SkillLibrary.search() with BM25-only (no embedding_provider) returns relevant results
    - SkillLibrary.search() with _FakeEmbedder returns hybrid-ranked results
    - SkillLibrary.add_or_merge() with near-duplicate skill triggers merge (heuristic path, no merge_llm)
    - SkillLibrary.add_or_merge() with distinct skill adds without merging
  </behavior>
  <action>
Create `tests/test_opengui_p1_skills.py`. At the top define shared test helpers:

1. `_FakeEmbedder` — same pattern as memory tests: returns deterministic float32 unit vectors, `async def embed(self, texts) -> np.ndarray`.

2. `_ScriptedLLM` — accepts canned response strings, returns `LLMResponse(content=response)` from `async def chat()`. Used for SkillExtractor tests in Task 2.

3. `_FakeValidator` — accepts `list[bool]`, pops and returns from `async def validate()`. Used for SkillExecutor tests in Task 2.

Write the following tests (use `tmp_path / "skills"` for each test to avoid `__post_init__` loading stale files — see Pitfall 2 in research):

Sync tests:
- `test_skill_library_crud`: Add skill, verify get() returns it, verify count==1, remove it, verify count==0
- `test_skill_library_persists_to_disk`: Add skill, create new SkillLibrary with same path, verify skill reloaded
- `test_skill_library_list_all`: Add 3 skills, verify list_all() returns all 3

Async tests:
- `test_skill_library_search_bm25_only`: Create library with no embedding_provider, add 3 skills with different descriptions, search for a term, verify most relevant skill is in results
- `test_skill_library_search_hybrid`: Create library with _FakeEmbedder, add skills, search, verify results returned
- `test_skill_library_dedup_merges_similar`: Add skill s1, then call add_or_merge(s2) where s2 has same name as s1 but updated description — verify decision is one of ("MERGE", "KEEP_OLD", "KEEP_NEW") and count stays 1
- `test_skill_library_dedup_adds_distinct`: Add skill s1, then add_or_merge(s2) with a completely different name — verify decision is "ADD" and count is 2

Always use a fresh `tmp_path / "skills_N"` subdirectory per test.

  </verify>
