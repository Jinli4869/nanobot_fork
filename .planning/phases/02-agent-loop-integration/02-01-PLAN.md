**MemoryStore migration (opengui/memory/store.py):**

Rewrite internals to use one `.md` file per MemoryType instead of `memory.json`:
- `os_guide.md`, `app_guide.md`, `icon_guide.md`, `policy.md`
- Each H2 section (`## heading`) = one MemoryEntry
- Format per section:
  ```
  ## {entry topic}
  entry_id: {entry_id}
  platform: {platform}
  app: {app}

  {content body}
  ```
- Add `_parse_md_file(text: str, memory_type: MemoryType) -> list[MemoryEntry]` helper using `re.compile(r"^## (.+)$", re.MULTILINE)` to split text into sections
- Add `_entry_to_md(entry: MemoryEntry) -> str` helper that formats one entry as an H2 section
- `save()` writes one `.md` file per type with all entries of that type concatenated
- `load()` reads each `.md` file and parses entries via `_parse_md_file()`
- Keep the same public API: `add()`, `remove()`, `get()`, `list_all()`, `count`
- Use atomic write pattern (tempfile + rename) same as current JSON version
- Remove `memory.json` path logic entirely

Metadata line parsing: lines matching `key: value` before the first blank line in a section body are metadata. After the first blank line, everything is content. If `entry_id:` metadata exists, use it; otherwise slugify the H2 heading text (lowercase, spaces to hyphens, strip non-alphanumeric).

**Memory prompt injection (opengui/prompts/system.py):**

In `build_system_prompt()`, wrap the memory_context in `<memory>` XML tags:
- Change: `sections.extend(["", "# Relevant Knowledge", "", memory_context])`
- To: `sections.extend(["", "# Relevant Knowledge", "", "<memory>", memory_context, "</memory>"])`

This satisfies the locked decision that memory entries are injected as a `<memory>` section in the system prompt.
